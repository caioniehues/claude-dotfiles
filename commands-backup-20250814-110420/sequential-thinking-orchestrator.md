<routing_strategies>
## Thinking Mode Routing

### Mode 1: Simple XML Thinking (Complexity ≤ 2)
<simple_mode_thinking>
When to use:
- Clear, well-defined problems
- Limited decision points
- Predictable outcomes
- Fast execution needed

Implementation:
```xml
<thinking>
Problem: [statement]
Approach: [strategy]
Solution: [implementation]
Validation: [check]
</thinking>
```

Advantages:
- Fast execution
- Low overhead
- Clear reasoning trace
- Easy to follow
</simple_mode_thinking>

### Mode 2: Hybrid Orchestration (Complexity = 3)
<hybrid_mode_thinking>
When to use:
- Moderate complexity
- Some uncertainty
- Multiple valid approaches
- Benefit from validation

Implementation:
1. Start with XML thinking blocks
2. Identify uncertainty points
3. Invoke MCP for specific decisions
4. Integrate results back
5. Continue with XML execution

Example flow:
```xml
<initial_thinking>
Basic analysis and approach
</initial_thinking>

<!-- Uncertainty point detected -->
INVOKE: MCP Sequential Thinking
FOR: Specific decision

<integration_thinking>
Incorporating MCP insights
</integration_thinking>

<execution>
Proceed with refined approach
</execution>
```
</hybrid_mode_thinking>

<orchestration_patterns>
## Orchestration Patterns

### Pattern 1: Sequential Escalation
<sequential_pattern>
Start simple, escalate as needed:

1. Begin with XML thinking
2. Monitor complexity indicators
3. If complexity increases:
   - Save XML context
   - Escalate to MCP
   - Include prior thinking
4. Return to XML for execution
</sequential_pattern>

### Pattern 2: Parallel Thinking
<parallel_pattern>
Use both simultaneously:

1. XML for main execution path
2. MCP for complex decisions
3. Synchronization points
4. Merged thinking output

Example:
- XML: Handle UI and simple logic
- MCP: Solve algorithm complexity
- Merge: Integrated solution
</parallel_pattern>

### Pattern 3: Recursive Orchestration
<recursive_pattern>
Nested thinking for sub-problems:

1. MCP identifies sub-problems
2. Each sub-problem assessed
3. Route to appropriate handler
4. Aggregate results
5. Return to parent thinking

Depth limit: 3 levels
Prevents infinite recursion
</recursive_pattern>

### Pattern 4: Checkpoint Orchestration
<checkpoint_pattern>
Thinking at decision gates:

1. Execute until checkpoint
2. Assess complexity at gate
3. Route to appropriate thinking
4. Make decision
5. Continue execution

Checkpoints:
- Major decision points
- Error conditions
- Validation gates
- Integration points
</checkpoint_pattern>
</orchestration_patterns>
